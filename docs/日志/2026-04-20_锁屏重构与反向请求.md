# 锁屏首页重构 + ESP → PC 反向请求模式

**日期：** 2026-04-20
**分支：** `feat/optimize_function`

## 任务概述

这次迭代围绕两个独立但互相配合的目标推进：

1. **UI 层**：把原本混在时间页里的"调节按钮 + 时间显示"拆开，让 `page_time` 变成纯展示的锁屏首页，调节功能搬到从菜单进入的新页面 `page_time_adjust`。同时用 LVGL 手势事件实现全屏上滑切菜单。
2. **通信层**：沉淀出一套"ESP 主动请求 PC 推数据"的通用模式，解决设备上电时间不准、进入天气页数据陈旧等问题；不新增 GATT Service、不切 BLE 角色。

两者叠起来的体验就是：**开机 → 锁屏显示时间 →（后台自动从 PC 同步时间）→ 上滑进菜单 → 点天气 →（后台自动从 PC 拉天气）→ 看到最新数据**，用户一次不用操心，BLE 协商全部静悄悄完成。

## 阶段一：锁屏首页重构

### 拆分前的问题

原 `page_time.c` 385 行，屏幕上是三张堆叠的卡片：

```
┌──────────────────────┐
│ 显示卡 (100px)       │  ← 时间 + 日期
├──────────────────────┤
│ 时间调节卡 (80px)    │  ← ±时 / ±分 按钮
├──────────────────────┤
│ 日期调节卡 (90px)    │  ← ±年 / ±月 / ±日 按钮
└──────────────────────┘
```

调节按钮占掉 2/3 的屏幕空间，作为首页观感很差。用户直觉："这应该像手机锁屏一样——只看时间。调节是高级功能，藏到二级页面。"

### 方案：一页拆两页

| 页面 | 职责 | 入口 |
|------|------|------|
| `page_time`（保留 ID `PAGE_TIME`） | 锁屏首页，只展示大号时间 + 中文日期 | 开机默认页 |
| `page_time_adjust`（新 ID `PAGE_TIME_ADJUST`） | 原来的时间/日期调节卡片 + Back 按钮 | 菜单 → Time & Date |

锁屏 → 菜单用 LVGL 手势；菜单 → 调节页用普通列表项点击；调节页 → 菜单用 Back 按钮。

### 手势触发：LVGL 9.5 的 `LV_EVENT_GESTURE`

关键发现（`managed_components/lvgl__lvgl/src/indev/lv_indev.c:1798-1815`）：LVGL 根据触摸位移累计判断方向，超过 `gesture_min_distance` 时向 screen 发 `LV_EVENT_GESTURE` 事件。只要 screen 上的子对象都不抢占点击（label 默认 `CLICKABLE=off`），手势就会冒泡到 screen。

实现只需三行：

```c
// page_time.c
static void on_gesture(lv_event_t *e) {
    lv_dir_t dir = lv_indev_get_gesture_dir(lv_indev_active());
    if (dir == LV_DIR_TOP) page_router_switch(PAGE_MENU);
}

// page_time_create 里
lv_obj_add_event_cb(s_ui.screen, on_gesture, LV_EVENT_GESTURE, NULL);
```

### 48px 大号字体——不动 sdkconfig 的取巧做法

锁屏要求时间字号足够大（240×320 屏上至少 48px）。两个方案：

| 方案 | 代价 |
|------|------|
| A. 开 `CONFIG_LV_FONT_MONTSERRAT_48` | 改 sdkconfig.defaults，要删 sdkconfig 重生成，固件多约 50KB 点阵 |
| B. 对已嵌入的 `srhs_sc_subset.ttf` 再 `lv_tiny_ttf_create_data_ex` 一个 48px 实例 | 零固件增加（TTF 已在），运行时 cache 只画 `0-9:` 共 11 个字形 |

选 B。`app_fonts.c` 里加：

```c
g_app_font_huge = lv_tiny_ttf_create_data_ex(
    srhs_ttf_start, ttf_size, 48, LV_FONT_KERNING_NONE, 32);
g_app_font_huge->fallback = &lv_font_montserrat_24;
```

思源黑体 SC 自带完整 Latin 字符集，画数字没问题。

### 分钟级刷新节流

锁屏不显示秒，所以 `update_display` 比较 `tm_min`，同一分钟直接 return：

```c
if (t.tm_min == s_ui.last_min) return;   // 绝大多数 tick 命中这里
s_ui.last_min = t.tm_min;
// ... snprintf + set_text ...
```

UI 任务跑 100Hz，按钟粒度刷新省掉每帧 snprintf + LVGL 脏区计算。

## 阶段二：ESP → PC 反向请求模式（核心）

### 动机

- **时间不准**：ESP 重启后只能读 NVS 里 5 分钟前的旧时间，或硬编码默认时间；PC 端如果不主动推，屏上时间一直停在旧值
- **天气陈旧**：进入天气页看到的是上次 PC 推送的快照，可能是几小时前的
- **用户想要**：ESP 掌握主动权——连上 PC 或进入天气页时，"自己"去问 PC 要最新数据

### 为什么不能让 ESP 主动连 PC

BLE 有**两组独立角色**，容易混淆：

| 层 | 角色 A | 角色 B |
|----|--------|--------|
| Link 层（连接管理） | Peripheral（广播、被连） | Central（扫描、发起连接） |
| GATT 层（数据收发） | Server（提供 characteristic） | Client（读写对端 char） |

当前架构：ESP = Peripheral + GATT Server / PC = Central + GATT Client。让 ESP 切 Central 或当 GATT Client 去读 PC，**都要求 PC 当 GATT Server**：
- Bleak（PC 端库）**不支持** Peripheral API
- Windows 原生 `GattServiceProvider` 需另写一大套代码，且电源策略会让 PC 广播间歇性熄火

工程上不现实，放弃。

### 解法：复用 control_service NOTIFY + REQUEST 事件 id

`control_service` 本来就是 ESP → PC 方向的 Notify 通道（屏上按钮上报 PC）。扩展它的事件 type 字段：

```c
// services/control_service.h
#define CONTROL_EVENT_TYPE_BUTTON  0
#define CONTROL_EVENT_TYPE_SLIDER  1
#define CONTROL_EVENT_TYPE_REQUEST 2   // 新增：ESP 请求 PC 推数据

#define CONTROL_REQUEST_TIME_SYNC  0
#define CONTROL_REQUEST_WEATHER    1
// 预留：MEDIA_REFRESH = 2 ...
```

`control_event_t` 结构体字节数不变（仍 8 字节），PC 端只需在 `on_notify` 里根据 `type` 路由。

### 为什么挂在 SUBSCRIBE 上升沿，而不是 CONNECT

BLE 规范：服务端 Notify 一个 characteristic 之前，客户端必须先**订阅**（往该 char 的 CCCD 写 `0x0001`）。PC 端的 `await client.start_notify(CONTROL_CHAR_UUID, ...)` 做的就是这件事。

ESP 这边收到 CCCD 写入时，不是触发 GATT access_cb，而是触发 GAP 层的 `BLE_GAP_EVENT_SUBSCRIBE` 事件。事件带 `attr_handle` / `prev_notify` / `cur_notify` 三个关键字段。

如果在 `BLE_GAP_EVENT_CONNECT` 就发 notify，此时 CCCD 还是 0，NimBLE 会把 notify 丢掉（`rc=6`）。**SUBSCRIBE 上升沿（`prev=0, cur=1`）是最早能安全发 notify 的时机**。

时序：

```
PC BleakClient.connect  ─────►  ESP BLE_GAP_EVENT_CONNECT
PC start_notify(control_char)
  = 往 CCCD 写 0x0001     ─────►  ESP BLE_GAP_EVENT_SUBSCRIBE
                                       prev=0 cur=1
                                       ├─► control_service_on_subscribe
                                       │     └─► send_request(TIME_SYNC)
                                       │           └─► ble_gatts_notify_custom
                                   ◄───┤
PC ControlHandler.on_notify 识别 REQUEST
  └─► push_cts(client)
        └─► write_gatt_char(CTS, now)
                                   ─────►  ESP time_service access_cb WRITE
                                              └─► time_manager_request_set_time
                                                    └─► xQueueSend
                                              (ui_task 下一帧消费 → settimeofday)
```

### 时间同步实现

**ESP 端**：

```c
// services/control_service.c
void control_service_on_subscribe(uint16_t attr_handle,
                                  uint8_t prev_notify, uint8_t cur_notify) {
    if (attr_handle != s_chr_val_handle) return;
    if (prev_notify || !cur_notify) return;        // 只处理上升沿
    control_service_send_request(CONTROL_REQUEST_TIME_SYNC);
}

// drivers/ble_driver.c: BLE_GAP_EVENT_SUBSCRIBE 分支调上面这个
```

**PC 端**（`tools/desktop_companion.py`）：

```python
async def push_cts(client):
    now = datetime.now()
    data = struct.pack(CTS_STRUCT, now.year, now.month, ..., now.isoweekday(), 0, 0)
    await client.write_gatt_char(CTS_CHAR_UUID, data, response=True)

# ControlHandler.on_notify 里：
if type_ == EVENT_TYPE_REQUEST:
    if btn_id == REQUEST_TIME_SYNC:
        asyncio.create_task(self._time_sync_cb())

# run_session 里连接成功时：
ctrl.set_time_sync_cb(lambda: push_cts(client))
```

实测：ESP 重启 → 约 2 秒后看到屏上时间跳到 PC 当前时间。

### 天气请求实现（同一套模板）

扩展成本极低：

**ESP 端**（2 个文件，4 行代码）：
- `control_service.h` 加 `CONTROL_REQUEST_WEATHER = 1`
- `page_weather.c` 的 `page_weather_create` 末尾加 `control_service_send_request(CONTROL_REQUEST_WEATHER)`

**PC 端**：
- 把 `ble_time_sync.py` 的 `Weather` dataclass / `WMO_DESC` / `locate_by_ip` / `fetch_weather` 搬到 `desktop_companion.py`
- 新增 `push_weather(client)`，内部调 `loop.run_in_executor` 异步化 `requests` 调用
- `ControlHandler` 加 `set_weather_cb` + `on_notify` 的 REQUEST 分支加 `REQUEST_WEATHER` 路由

### PC 端 10 分钟缓存

天气 API 是公开免费服务（open-meteo + ip-api），频繁调用容易被限流。用户可能反复进出天气页，每次都打 API 不友好。

```python
_weather_cache: Optional[tuple[float, Weather]] = None
_weather_location: Optional[tuple[float, float, str]] = None
WEATHER_CACHE_TTL_S = 600

async def push_weather(client):
    global _weather_cache, _weather_location
    now_ts = _time.time()
    if _weather_cache and now_ts - _weather_cache[0] < WEATHER_CACHE_TTL_S:
        w = _weather_cache[1]
        log(f"[weather] cache hit ({int(now_ts - _weather_cache[0])}s old)")
    else:
        if _weather_location is None:
            _weather_location = await loop.run_in_executor(None, locate_by_ip)
        lat, lon, city = _weather_location
        w = await loop.run_in_executor(None, fetch_weather, lat, lon, city)
        _weather_cache = (now_ts, w)
    await client.write_gatt_char(WEATHER_CHAR_UUID, w.pack(), response=True)
```

- `_weather_cache`：10 分钟过期
- `_weather_location`：进程生命周期内永不过期（桌面机地理位置不会变）
- 两级缓存 → 频繁进天气页只打 BLE write，不打 HTTP

## 阶段三：线程安全复盘

这次增量小但涉及跨线程调用，顺手复核了整个项目的线程模型。

### 当前线程分工

```
┌──────────────────────────┐        ┌─────────────────────────────────┐
│   ui_task (prio=5, 8KB)  │        │   BLE host task (NimBLE)        │
│                          │        │                                 │
│   lv_timer_handler        │        │   gap_event_handler             │
│    └─ LVGL 事件 dispatch  │        │    ├ CONNECT / DISCONNECT       │
│       └─ 按钮 / 手势回调  │        │    └ SUBSCRIBE（触发反向请求）│
│          └─ 调 NimBLE API │◄───────┤                                 │
│                          │ NimBLE │   GATT access callbacks         │
│   *_manager_process_     │ host   │    ├ time_service   access_cb   │
│   pending (出队更新快照) │ mutex  │    ├ weather_service access_cb  │
│                          │◄───────┤    └ ...                        │
│                          │ Queue  │                                 │
└──────────────────────────┘        └─────────────────────────────────┘
```

### 两条数据路径的保证

| 路径 | 机制 |
|------|------|
| UI → BLE：调 `ble_gatts_notify_custom` 等 | NimBLE 内部 host mutex，任意线程安全 |
| BLE → UI：收数据 | BLE 线程 `xQueueSend` → UI 线程 `xQueueReceive`，FreeRTOS Queue 本身线程安全 |
| UI 线程读 `manager_get_latest()` | 单写者（UI 线程自己）+ 同线程读，无锁 |
| UI 线程调 `lv_*` | LVGL 只在 ui_task 里调，符合单线程约束 |

**红线**：BLE 回调（GAP 或 GATT）里永远只能 push 队列或调 NimBLE API，**不能碰 LVGL、不能阻塞、不能写 NVS**。

### 发现的一个"有味道但不紧急"的点

`control_service.c` 的 `s_seq` 全局变量有两个线程会 `++`：
- UI 线程：按钮 / 进入天气页
- BLE host 线程：SUBSCRIBE 时发 TIME_SYNC 请求

`++s_seq` 编译成 LOAD + ADD + STORE 三条指令，两个线程交错时可能撞到同一个 seq。

实际后果：PC 端收到两条 seq 相同的 notify，去重逻辑会丢掉第二条。触发概率极低（需要两个线程在几十纳秒窗口内同时执行），现实中一年遇不到一次，**本次不修**。未来需要严格单调序号时改成 `atomic_fetch_add(&s_seq, 1)` 即可。

## 踩坑

### 1. 改了 Python 代码没重启进程

现象：ESP 日志显示 `request sent: id=1 seq=2`，PC 端日志显示 `[warn] 未知 request id=1`，但同一次连接的 `REQUEST_TIME_SYNC` 能正常识别。

原因：PC 端 `desktop_companion.py` 已经在跑，我改完文件后 Python 不会热重载，内存里加载的还是没加天气分支的旧版本。

解决：Ctrl+C 停掉重启。教训：**以后每次改 PC 端 .py 都要手动重启进程**；可以在 `main()` 开头加一行 `log("companion vN")` 便于肉眼辨别版本。

### 2. Python 同步 `requests` 库在 asyncio 里会卡住

`fetch_weather` / `locate_by_ip` 都用 `requests.get()`，这是同步阻塞调用。如果直接 `requests.get(url)` 放在 async 函数里，整个 event loop 会卡 10-15 秒，BleakClient 收 notify 全停摆。

解决：`await loop.run_in_executor(None, fetch_weather, lat, lon, city)` 丢到线程池，async 侧 await 等结果，event loop 期间继续处理其他 IO。不引入 `aiohttp` 少一个依赖。

### 3. 手势事件与按钮点击的冲突预期

担心：如果手指按在按钮上滑动，按钮会不会拦截手势？

实测：锁屏上没有按钮（纯 label），手势始终冒泡到 screen；菜单页没挂 GESTURE 回调，列表滚动是另一条 `LV_EVENT_SCROLL` 路径，互不干扰。不需要额外处理 `LV_OBJ_FLAG_GESTURE_BUBBLE`。

## 模式沉淀

这次确立的"**ESP 想要 PC 推 X → 扩展 `CONTROL_REQUEST_X` id**"是本项目所有反向请求的标准做法。下一次如果想让 ESP 进入音乐页时主动要媒体状态重推、或者请求 PC 重发通知列表，都可以无脑复用：

1. `control_service.h` 加 `CONTROL_REQUEST_XXX = N`
2. 触发点调 `control_service_send_request(CONTROL_REQUEST_XXX)`
3. PC 端 `ControlHandler` 加 `set_xxx_cb` + `on_notify` REQUEST 分支加 id 路由
4. `run_session` 里注入 `set_xxx_cb(lambda: push_xxx(client))`

不新建 GATT Service、不切 BLE 角色、不引 WiFi。低功耗预算守得住。
