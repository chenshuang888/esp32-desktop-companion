# 反向请求模式重构：从集中式 control 拆到各业务 service

**日期：** 2026-04-21
**分支：** `feat/treill_init`
**前置：** `docs/锁屏重构与反向请求模式日志.md`（v1 集中式方案）

## 任务概述

上一轮（2026-04-20）沉淀的"ESP → PC 反向请求"模式是：在 `control_service` 的 NOTIFY char 上复用 `type=REQUEST + id=<req>` 来传信号，PC 收到后用对应业务 service 的 WRITE char 回数据。当时只有 TIME_SYNC 一个请求，方案自洽；到这一轮补上 WEATHER、SYSTEM 共 3 个 REQUEST 后，设计撕裂开始暴露：

- **触发端 / 响应端跨 service**：信号由 `control_service` notify 发出，响应却落到 `time_service` / `weather_service` / `system_service` 各自的 write char —— 读代码要来回跳。
- **`control_service` 名实漂移**：这个文件叫 "control" 但已承担三种不同语义（按钮 + 反向请求枚举），新同事打开 `control_service.h` 会困惑。
- **`time_service.c` 早就声明了 NOTIFY flag**：原始设计本来就是各 service 自管，是上一轮临时借用 `control_service` 抄近路。

本轮重构：**反向请求回归各业务 service 自管**，`control_service` 回到纯按钮通道。

---

## 核心决策（开工前 3 个选项）

| 决策点 | 选择 | 为什么 |
|---|---|---|
| time 的反向请求通道 | 复用标准 CTS `0x2A2B` 的 NOTIFY flag（同 char WRITE+NOTIFY） | 不新增 UUID；CTS 本来就允许 NOTIFY（server 主动通知时间变化），语义兼容 |
| 是否抽 `ble_notify_helper` | 不抽，每个 service 自写 | 每个 service 多 15 行代码，抽 helper 只省 10 行；奥卡姆剃刀 |
| NOTIFY body 内容 | 1 字节递增 seq | 非空 body 避开某些 BLE 栈对 zero-length notify 的奇怪处理；日志便于排查 |

## UUID 分配（打破"service 内奇偶成对"规则）

原项目规则：每个自定义 service 占一对连号短码（奇=service / 偶=characteristic）。但此次需要给已落地的 weather / system 各补一个 NOTIFY req char —— service 内 char 超过 2 个，连号段早被下一个 service 占走了。

采用**末尾追加**例外：

| 短码 | 角色 | 方向 | payload |
|---|---|---|---|
| `0x8a5c0001` | weather service | — | — |
| `0x8a5c0002` | weather write char | PC → ESP (WRITE) | 68B packed |
| `0x8a5c0009` | system service | — | — |
| `0x8a5c000a` | system write char | PC → ESP (WRITE) | 16B packed |
| `0x8a5c000b` | **weather req char** | ESP → PC (NOTIFY) | 1B seq |
| `0x8a5c000c` | **system req char** | ESP → PC (NOTIFY) | 1B seq |

time 不占新 UUID，直接复用 `0x2A2B` 的 NOTIFY flag。

该例外已写入 `.trellis/spec/iot/protocol/ble-custom-uuid-allocation-decision-record.md`，规则：**初次设计时没预留额外 char 的 service，事后可在整张表末尾追加新 UUID；文件顶部注释和 DR 表格双重说明归属**。

## ESP 固件改动

### 每个业务 service 的新接口

统一暴露两个公共函数：

```c
esp_err_t <svc>_service_send_request(void);   // UI/订阅钩子调用
void      <svc>_service_on_subscribe(uint16_t attr, uint8_t prev, uint8_t cur);
```

`_send_request` 实现：取 `ble_conn_get_handle`、`++s_req_seq`、组 1B mbuf、`ble_gatts_notify_custom`。
`_on_subscribe` 实现：匹配本服务 val_handle、判上升沿（`prev==0 && cur==1`）、调 `_send_request`。

### time_service 特别处理

删除无人调用的 `time_service_notify_time_changed()`（原设计是"server 通知 client 时间变了"，新语义正好反过来，保留会混淆）。GATT flag 保持 `READ | WRITE | NOTIFY` 不变，只是 `current_time_val_handle` 现在同时被 write handler 和 notify 用。

### control_service 瘦身

删除 `CONTROL_EVENT_TYPE_REQUEST` / `CONTROL_REQUEST_*` 宏、`control_service_send_request()`、`control_service_on_subscribe()`。header 顶部注释改为"ESP → PC 按钮事件通道"，功能单一化。

### ble_driver 订阅广播

原来 SUBSCRIBE 分支只调 `control_service_on_subscribe`。改为：

```c
case BLE_GAP_EVENT_SUBSCRIBE:
    time_service_on_subscribe(attr, prev, cur);
    weather_service_on_subscribe(attr, prev, cur);
    system_service_on_subscribe(attr, prev, cur);
    break;
```

每个 service 自己判"是不是我的 char"，不匹配就直接 return。

### 页面触发点切换

| 位置 | 变更 |
|---|---|
| `app/pages/page_weather.c:327` | `control_service_send_request(CONTROL_REQUEST_WEATHER)` → `weather_service_send_request()` |
| `app/pages/page_system.c:378` | `control_service_send_request(CONTROL_REQUEST_SYSTEM)` → `system_service_send_request()` |

header include 也同步从 `control_service.h` 换成 `<svc>_service.h`。

## PC 端改动（`tools/desktop_companion.py`）

### 前后对比

```text
v1（集中式）                       v2（自管）
────────────────────────────────   ────────────────────────────────
start_notify(CONTROL)              start_notify(CONTROL)   // 只管按钮
  └─ ControlHandler.on_notify      start_notify(CTS)       → push_cts
      ├─ type==BUTTON → keybd      start_notify(WEATHER_REQ) → push_weather
      └─ type==REQUEST             start_notify(SYSTEM_REQ)  → sys_pub.push_now
          ├─ id==TIME → cb1
          ├─ id==WEATHER → cb2
          └─ id==SYSTEM → cb3
       + 三个 set_*_cb 注入点        + 一个 make_request_handler 工厂
```

### 关键组件

```python
def make_request_handler(name: str, action):
    def handler(_sender, data: bytearray):
        seq = data[0] if data else None
        log(f"[req] {name} seq={seq}")
        asyncio.create_task(action())
    return handler
```

`run_session` 在每个 req char 上独立 `start_notify`，handler 直接调 `push_cts` / `push_weather` / `sys_pub.push_now`。`ControlHandler` 瘦身为只处理按钮事件（删掉 `set_time_sync_cb` / `set_weather_cb` / `set_system_cb` 三个注入点、`EVENT_TYPE_REQUEST` / `REQUEST_*` 常量、未用的 `Awaitable/Callable` import）。

## 完整的出方向通知通道（重构后）

### 主动 NOTIFY

| 场景 | 函数 | 位置 | UUID |
|---|---|---|---|
| 控制页按钮按下 | `control_service_send_button(id)` | `app/pages/page_control.c:207` | `8a5c0006` (8B `control_event_t`) |
| 音乐页播放/暂停/上下曲按下 | `control_service_send_button(id)` | `app/pages/page_music.c:305` | 同上 |
| CTS 被订阅上升沿 → 请 PC 推时间 | `time_service_send_request()` | `services/time_service.c:237` | 标准 `0x2A2B` NOTIFY (1B seq) |
| weather-req 被订阅 → 请 PC 推天气 | `weather_service_send_request()` | `services/weather_service.c:162` | `8a5c000b` (1B seq) |
| 进入天气页 → 请 PC 推天气 | `weather_service_send_request()` | `app/pages/page_weather.c:327` | 同上 |
| system-req 被订阅 → 请 PC 推系统帧 | `system_service_send_request()` | `services/system_service.c:161` | `8a5c000c` (1B seq) |
| 进入系统页 → 请 PC 推系统帧 | `system_service_send_request()` | `app/pages/page_system.c:378` | 同上 |

### 被动 ATT response（不是独立通知）

PC `read` / `write` 时 ESP 在 `access_cb` 里返回 —— 随请求走，不能独立发起：
- READ → ESP 在 `os_mbuf_append` 塞数据（CTS 返当前时间；各 req char 返最新 seq 作心跳占位）
- WRITE → ESP return 0 或 `BLE_ATT_ERR_*` 错误码（PC 侧 `response=True` 时可见）

**本项目未用**：INDICATE（NOTIFY 需 ACK 版）、Service Changed、WRITE 回调里额外触发 NOTIFY 做 ack。

## 未来新增一个反向请求业务（模板）

例：假设要做"进入音乐页让 PC 重推媒体状态"。

1. `services/media_service.c` GATT 表加第二个 char，UUID 继续末尾追加（下一个空位 `0x8a5c000d`），flag `READ | NOTIFY`
2. 新增 `media_service_send_request()` + `media_service_on_subscribe()`
3. `drivers/ble_driver.c` SUBSCRIBE 广播里加一行 `media_service_on_subscribe(...)`
4. `tools/desktop_companion.py` 新增 `MEDIA_REQ_CHAR_UUID`，`run_session` 里：
   ```python
   await client.start_notify(
       MEDIA_REQ_CHAR_UUID,
       make_request_handler("media", pub.push_now),
   )
   ```
5. 页面 enter 调 `media_service_send_request()`

不动既有代码，新业务 ≤50 行 C + 4 行 Python 搞定。这是本次重构想要达到的"加业务不改老代码"的目标。

## 一句话总结

**触发端与响应端必须同在一个 service 内。** 这是本轮学到的最重要的跨层设计原则，其他所有变更都是它的推论。
