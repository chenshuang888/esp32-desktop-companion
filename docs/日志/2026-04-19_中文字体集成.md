# 中文字体集成开发日志

**日期：** 2026-04-19
**分支：** `feat/achieve_chinese`

## 任务概述

屏幕上所有中文字段（通知 title/body、歌名/歌手、城市/天气描述）全是方块麻将。LVGL 默认只编入 Montserrat 英文字体，不含 CJK 字形。需要让 ESP32-S3 能显示常用中文而又不撑爆资源。

**目标：**
- PC 端推送的任意常用中文都能正确渲染
- 保持 `app/services/drivers` 分层，字体是 UI 资源只放 `app/`
- 所有页面代码（`page_*.c`）不动 —— 改字体方案不应波及业务代码
- Flash 占用可控（已在 16MB N16R8，但要为未来功能留余量）

最终方案迭代了三轮：**LVGL 内置位图 → 霞鹜文楷 TTF → Noto Sans SC TTF**。

---

## 阶段一：首选 LVGL 内置 CJK 字库

LVGL 9.5 本身自带两档简体中文字体：`lv_font_source_han_sans_sc_14_cjk` / `_16_cjk`。sdkconfig 勾一下就能用，不需要外部工具链。

### 改动内容

- **sdkconfig.defaults**：开启 `CONFIG_LV_FONT_SOURCE_HAN_SANS_SC_14/16_CJK=y`
- **app/app_fonts.{h,c}**（新建）：定义宏别名 + 挂 fallback

```c
// app_fonts.h
extern lv_font_t g_app_font_text;       // 14px 正文
extern lv_font_t g_app_font_title;      // 16px 标题
#define APP_FONT_TEXT   (&g_app_font_text)
#define APP_FONT_TITLE  (&g_app_font_title)
#define APP_FONT_LARGE  (&lv_font_montserrat_24)   // 大号数字沿用 Montserrat
void app_fonts_init(void);

// app_fonts.c
void app_fonts_init(void) {
    g_app_font_text          = lv_font_source_han_sans_sc_14_cjk;  // 值拷贝
    g_app_font_text.fallback = &lv_font_montserrat_14;
    g_app_font_title          = lv_font_source_han_sans_sc_16_cjk;
    g_app_font_title.fallback = &lv_font_montserrat_20;
}
```

- **8 个页面 + app_ui.c**：批量替换 ~44 处 `&lv_font_montserrat_14/20/24` → `APP_FONT_TEXT/TITLE/LARGE`

### 坑 1：LVGL 内置字体是 const，不能直接改 fallback

```c
// lv_font_source_han_sans_sc_14_cjk.c 里
const lv_font_t lv_font_source_han_sans_sc_14_cjk = { ... };
```

直接 `lv_font_source_han_sans_sc_14_cjk.fallback = xxx` 会编译失败。**解决**：把整个结构体值拷贝到 `.bss` 段的可写副本 `g_app_font_text`，再改副本字段。后来切 Tiny TTF 时这一层就不需要了（工厂函数返回的本来就是可写堆对象）。

### 发现问题：1118 字根本不够

验证时 PC 推送一条 `title="紫罗兰永恒花园"` 的通知，结果"紫"、"罗"、"兰"都还是方块。查 LVGL 内置字体源文件头部 `--symbols` 参数，用 Python 数了一下：

```
CJK 字符数: 1118 (去重)
日文假名: 150
前 20 字: 盗提陽帯鼻画輕冊写父結想正四夫源庭場天續
```

**1118 字**，而且还掺杂日文假名和繁体字，简体常用字覆盖严重不足。GB2312 一级字符集是 3755，常用 3500 字表也差不多，LVGL 内置只有约三分之一。对一个要承接"PC 任意消息"的 UI 设备，这个覆盖面直接出局。

## 阶段二：改用 Tiny TTF 运行时渲染

既然预生成字库字数硬性不够，就该走**矢量字体运行时栅格化**路线。调研了三条路：

| 方案 | 字符覆盖 | Flash | RAM | 工具链 |
|---|---|---|---|---|
| `lv_font_conv` 预生成 GB2312 3755 字（C 数组）| 常用 99% | ~900 KB | 0 | Node.js + lv_font_conv |
| **Tiny TTF + 子集 TTF 嵌固件** | **任意字** | **~1.5 MB TTF** | ~120 KB cache | pyftsubset 一次 |
| FreeType + partition TTF | 任意字 | TTF 独立分区 | ~100 KB | 最重 |

**选 Tiny TTF**：
- LVGL 9.5 原生支持（`src/libs/tiny_ttf/`，Kconfig 暴露 `LV_USE_TINY_TTF`），无需引入 FreeType 组件
- 单一 TTF 文件供所有字号用（14/16 共享一份数据）
- 字符覆盖不受预生成字库限制，PC 推什么字都画得出
- 前提：PSRAM 必须开启 —— glyph cache 吃 ~100 KB，落内部 RAM 撑不住

架构上字体层面向上层的接口不变，APP_FONT_* 宏同名不同物：

```c
// 方案 A：结构体（值拷贝 + 改 fallback）
lv_font_t g_app_font_text;             // .bss
#define APP_FONT_TEXT (&g_app_font_text)

// 方案 B：指针（工厂函数返回堆对象）
lv_font_t *g_app_font_text;            // 指针
#define APP_FONT_TEXT (g_app_font_text)
```

页面代码里 `lv_obj_set_style_text_font(lbl, APP_FONT_TEXT, 0)` 一行都不改——两种方案都是 `lv_font_t *`，LVGL 通过字体里的 3 个函数指针自动分派到不同后端。**这就是 C 语言版的接口多态**：把字形底层换了，页面感知不到。

### 新版 app_fonts.c

```c
extern const uint8_t srhs_ttf_start[] asm("_binary_srhs_sc_subset_ttf_start");
extern const uint8_t srhs_ttf_end[]   asm("_binary_srhs_sc_subset_ttf_end");

lv_font_t *g_app_font_text  = NULL;
lv_font_t *g_app_font_title = NULL;

void app_fonts_init(void) {
    const size_t ttf_size = srhs_ttf_end - srhs_ttf_start;
    g_app_font_text = lv_tiny_ttf_create_data_ex(
        srhs_ttf_start, ttf_size, 14, LV_FONT_KERNING_NONE, 256);
    g_app_font_title = lv_tiny_ttf_create_data_ex(
        srhs_ttf_start, ttf_size, 16, LV_FONT_KERNING_NONE, 256);
    g_app_font_text->fallback  = &lv_font_montserrat_14;
    g_app_font_title->fallback = &lv_font_montserrat_20;
}
```

- `cache_size=256`：每字号 cache 256 个 glyph 位图，够盖住项目活跃字集
- `LV_FONT_KERNING_NONE`：中文基本不用 kerning，关掉省 cache 空间
- fallback 链：CJK 字体不含 `LV_SYMBOL_*`（FontAwesome 私有区），fallback 到 Montserrat 画图标

### CMakeLists 配合

```cmake
idf_component_register(
    SRCS "app_main.c" "app_fonts.c" "pages/*.c"
    EMBED_FILES "fonts/srhs_sc_subset.ttf"   # ← 新增
    INCLUDE_DIRS "."
    REQUIRES framework drivers services lvgl
)
```

EMBED_FILES 把 TTF 原样转成汇编 `.S`，生成 `_binary_srhs_sc_subset_ttf_start/end` 符号，链接进 flash `.rodata` 段。不走文件系统，也不用 mount。

### sdkconfig 切换

```
CONFIG_LV_USE_TINY_TTF=y
CONFIG_LV_TINY_TTF_CACHE_GLYPH_CNT=256
CONFIG_LV_TINY_TTF_CACHE_KERNING_CNT=0
# CONFIG_LV_FONT_SOURCE_HAN_SANS_SC_14_CJK is not set   ← 关掉省 ~800KB
# CONFIG_LV_FONT_SOURCE_HAN_SANS_SC_16_CJK is not set
CONFIG_LV_USE_CLIB_MALLOC=y    # LVGL 走 libc malloc → heap_caps 自动落 PSRAM
```

## 阶段三：霞鹜文楷方案的失败

用户偏好单 TTF 直链下载，我先选了 **霞鹜文楷 Regular**（`github.com/lxgw/LxgwWenKai/releases`，v1.510，19.3MB TTF）。

### 子集化流程

写了 `tools/gen_font_subset.py`：
1. 自动扫 `app/fonts/` 找源 TTF
2. 生成 GB2312 全集字符表（`gb2312` 编码区间 0xA1A1-0xFEFE 解码 + ASCII + CJK 标点，共 7726 字符）
3. 调 `fontTools.subset`（pyftsubset）生成子集
4. drop 一堆无用表（DSIG/vhea/vmtx/FFTM/PfEd/TSI0-5/hdmx/LTSH）
5. 产物 `srhs_sc_subset.ttf`

pyftsubset 命令核心参数：
```
--text-file=字符集文件
--no-hinting            # stb_truetype 不用 hinting
--no-glyph-names        # 省 post 表空间
--desubroutinize        # CFF 友好（虽然 TTF 不用 CFF）
--legacy-cmap           # 保证 stb_truetype 能读
```

### 产物大小：3.4 MB —— 撞墙

```
[INFO] 源字体: LXGWWenKai-Regular.ttf (18.4 MB)
[INFO] 收录字符数: 7726
[OK] 生成 srhs_sc_subset.ttf  (3394.8 KB)
```

3.4 MB + 现有固件 ~1.3 MB = **~4.7 MB 估算**，直接撞破 app 分区 4MB 天花板。

### 诊断：glyf 占 97%

```
Total: 3394.6 KB
  glyf    3,383,902 B   97.3%
  loca       30,840 B    0.9%
  hmtx       30,724 B    0.9%
  cmap       29,184 B    0.8%
  其他         ~1 KB    < 0.1%
```

**glyf 才是大头**，继续 drop 其他表最多省 1KB，无感。7726 字 ÷ 3.38 MB = **每字 445 字节**。楷体笔画多、曲线 Bezier 节点密集，这是字形特征的物理下限。

结论：**同字数 + 楷体风格 + <1.5MB 是不可能三角**。必须换字体类型。

## 阶段四：Noto Sans SC 黑体替换

黑体每字 Bezier 节点远少于楷体，数据密度只有楷体的 ~40%。

### 下载源

查了几个直链：
- `notofonts/noto-cjk` 官方 repo 的 SubsetTTF 路径 404（目录结构与 WebSearch 结果不符）
- GitHub API 查目录被 rate limit
- 最终用 **`jsntn/webfonts` 镜像仓的直链**：
  `https://github.com/jsntn/webfonts/raw/master/NotoSansSC-Regular.ttf`

下载后 `file` 命令验证：**TrueType Font data**（不是 CFF/OTF，stb_truetype 能处理）。大小 2.07 MB ——**本身就是子集版**（不是完整 Noto CJK）。

### 字符覆盖验证

担心 2MB 的 Noto 字数不够，先用 Python 检查：

```
Total glyphs with unicode mapping: 8834
CJK Unified (U+4E00-U+9FFF): 7832
GB2312 全集缺字: 241
前 30 缺字: ・―ΑΒΓΔΕΖΗΘΙΚΛΜΝΞΟΠΡΣΤΥΦΧΨΩαβγδ
```

缺的 241 字全是**希腊字母 / 罗马数字 / 特殊符号**，项目 UI 完全不会出现。**CJK 汉字 7832 个全覆盖 GB2312 全集**，实用等价完整。

### 子集化结果：1.57 MB

```
[INFO] 源字体: NotoSansSC-Regular.ttf (2.1 MB)
[INFO] 收录字符数: 7726
[OK] 生成 srhs_sc_subset.ttf  (1606.0 KB)
```

**从 3.4 MB 压到 1.57 MB**（字数不变）。

进一步查表分布：

```
Total: 1605.8 KB
  glyf    1,561,521 B   95.0%
  其他       ~44 KB     5.0%
```

7726 字 ÷ 1.56 MB = **每字 ~200 字节**，是楷体（445 B）的 45%。再压就要失真（降低坐标精度会让屏幕上笔画变"生硬"），不值得折腾。

### 1.57MB vs 用户目标 1.5MB：超 70KB 的 trade-off

用户原本希望 <1.5MB。实际差 70KB（4.5%）。给了三条选项：

- **接受 1.57MB** ← 最终选择。app 分区 6MB 下固件估算 ~2.3 MB，占 38%，余量 3.7MB
- 换阿里巴巴普惠体（要手工官网表单下载，产物可能 ~1.3 MB）
- 砍字到 4500 常用（违反"不减字数"）

用户拍板接受。这个 70KB 换来了"不用再折腾字体"。

### 配合：app 分区 4 → 6 MB

在选霞鹜失败的时候就改过一次 partitions.csv：

```
# ESP32-S3 N16R8 partition layout (16MB)
nvs,         data, nvs,     0x9000,   0x6000,
phy_init,    data, phy,     ,         0x1000,
factory,     app,  factory, 0x10000,  0x600000,   # 4MB → 6MB
storage,     data, spiffs,  ,         0x9EF000,   # ~12MB → ~9.9MB
```

storage 从 ~12MB 降到 9.9MB，未来做文件系统/OTA 仍绰绰有余。

## 阶段五：其他踩坑

### 坑 2：stb_truetype 不支持 OTF/CFF outlines

原本想用思源黑体 SC（Adobe 官方），下载下来是 `.otf`（CFF outlines 格式）。Tiny TTF 底层是 `stb_truetype.h`，**只支持 TrueType glyph outlines**，CFF 完全解析不了。

**解决**：字体源必须是 `.ttf`（TrueType）。思源黑体有第三方做的 TTF 转换版，但没那么主流。最终用 Noto Sans SC（有官方 TTF 版本）规避这个问题。

### 坑 3：sdkconfig.defaults 改了必须删 sdkconfig 才生效

ESP-IDF 的配置优先级：**sdkconfig 存在时以它为准，sdkconfig.defaults 只在 sdkconfig 缺失时作为初始模板**。新增 Kconfig 项（比如 `CONFIG_LV_USE_TINY_TTF`、`CONFIG_SPIRAM_*`）在旧 sdkconfig 里压根没有，reconfigure 也不会凭空加。

**规范流程**：
```bash
rm sdkconfig
idf.py fullclean
idf.py build     # 此时从 defaults 重新生成 sdkconfig
```

这个坑之前 NVS 日志和 PSRAM 配置都踩过，已经写进 memory 了。

### 坑 4：Tiny TTF 必须搭 PSRAM

glyph cache 每字号 ~60 KB（256 项 × ~200B/项含位图），两档字号 ~120 KB。若走内部 RAM（总 ~320KB）会吃掉 37%，和 LVGL 其他对象、BLE 协议栈一起撑爆。

**配套 sdkconfig**（已在 PSRAM 日志里配好）：

```
CONFIG_SPIRAM=y
CONFIG_SPIRAM_MODE_OCT=y
CONFIG_SPIRAM_USE_MALLOC=y
CONFIG_SPIRAM_MALLOC_ALWAYSINTERNAL=16384    # >16KB 自动落 PSRAM
CONFIG_LV_USE_CLIB_MALLOC=y                  # LVGL 走 libc malloc → heap_caps
CONFIG_BT_NIMBLE_MEM_ALLOC_MODE_EXTERNAL=y   # BLE 堆也走 PSRAM
```

中文字体这轮的顺利推进，**前一轮 PSRAM 配置是基础**。没有 PSRAM 这条路根本走不通。

### 坑 5：EMBED_FILES 的符号命名

ESP-IDF 的 `EMBED_FILES "fonts/srhs_sc_subset.ttf"` 生成的符号是：

```
_binary_srhs_sc_subset_ttf_start
_binary_srhs_sc_subset_ttf_end
```

规则：只用 **basename**（去掉路径），把 `.` 换成 `_`。如果未来改文件名，app_fonts.c 里 extern 声明要同步改。

## 架构决策

### 字体层只放 app/，不下沉

`app/app_fonts.{h,c}` 归属 UI 层。services 不知道字体存在，drivers 也不调字体 API。`APP_FONT_*` 宏由 app 定义，页面通过 `#include "app_fonts.h"` 使用。

**不这么做的话会出问题**：若字体放 drivers/lvgl_port.c 里，services 如果要生成"带字形的位图"（比如未来的状态栏预渲染），会形成 services → drivers 的反向依赖（drivers 已 REQUIRES services）。字体是 UI 资源，放 UI 层最自然。

### 接口多态：切换字体后端不波及上层

`lv_font_t` 有 14 个字段，其中 3 个是函数指针（`get_glyph_dsc` / `get_glyph_bitmap` / `release_glyph`）、1 个是 `dsc` 隐藏私有数据。LVGL 内置位图字体 vs Tiny TTF 在这 4 个字段里填完全不同的实现，但**对外暴露的都是 `lv_font_t *`**。

`lv_obj_set_style_text_font(lbl, font_ptr, 0)` —— 任何符合 `lv_font_t *` 契约的对象都能塞进来。底层从"内置位图 const 结构体拷贝"切到"Tiny TTF 工厂函数 malloc"，**页面代码一个字符都不改**。这个抽象是 LVGL 架构里最关键的设计之一。

### 工厂函数的所有权契约

Tiny TTF 的 `lv_tiny_ttf_create_data_ex()` 内部：
1. `lv_malloc` 私有 `ttf_font_desc_t`（stbtt 状态 + 三个 cache 句柄）
2. `lv_malloc` `lv_font_t` 本体
3. 挂上函数指针 + `.dsc = 私有结构体`
4. 返回 `lv_font_t *`

**销毁必须调 `lv_tiny_ttf_destroy(font)`** —— 它会顺 `.dsc` 释放 cache、stbtt、lv_font_t 三样。不能直接 `lv_free(font)`，会漏 cache 和私有结构体。

我们项目里字体一旦 init 就用到关机，没调 destroy。架构上是悬空的一个 "TODO" —— 若未来做动态字体切换，要补这个释放。

## 经验教训

1. **LVGL 内置 CJK 字体是个陷阱**。看起来"sdkconfig 勾一下就有中文"很吸引人，实际只有 1118 字（还带日文/繁体），对"承接 PC 任意消息"的应用覆盖不足。LVGL 这套内置字库更适合固定词汇 UI（十几二十个按钮 label），不适合动态文本。第一轮选这条路是走了弯路。

2. **矢量字体的 glyf 是物理下限**。`pyftsubset` 能 drop 无用表，但 glyph outline 数据本身是字形信息的**信息论极限**。同字数下想再压只能换更紧凑的字体风格（黑体 < 宋体 < 楷体），或降低坐标精度（失真代价）。**"楷体 + 7726 字 + 1.5MB"是不可能三角**。

3. **TTF 必须是 TrueType outlines**，不是所有 `.ttf` 后缀都是。Adobe 的思源黑体发的是 `.otf`（CFF outlines），stb_truetype 解析不了。看起来两种都是 "TTF"，底层 outline 表示完全不同。选字体源时用 `file xxx.ttf` 验证 `TrueType Font data` 字样。

4. **GitHub API 容易被 rate limit**，探目录不如直接 `curl -I` 探直链 HTTP 状态码。`jsntn/webfonts` 这种镜像仓反而比官方 `notofonts/noto-cjk` 好下——直链稳定、单文件、无目录嵌套。

5. **Tiny TTF 和 PSRAM 是绑定关系**。没有 PSRAM 的项目硬上 Tiny TTF 会在 init 阶段 OOM，因为 glyph cache 随便就 100KB+。本项目能走通这条路是因为之前 N16R8 的 PSRAM 配置先就位了。做嵌入式架构决策时要意识到这种**依赖链**：字体方案 → RAM 容量 → PSRAM 开关 → sdkconfig 组合。

6. **EMBED_FILES 比 spiffs 分区简单得多**。当 TTF 不到 2MB 且不需要动态替换时，直接编进固件（`.rodata`）是最省事的做法：
   - 不用 `idf.py spiffs_image` 多一步
   - 不用 `LV_TINY_TTF_FILE_SUPPORT=y` + 运行时 mount
   - 烧录只要一个 `idf.py flash`
   代价是 TTF 和固件绑死，换字体要重烧。对"跑通优先"的项目合适，真有动态字体需求再迁 spiffs。

7. **接口多态是字体方案可切换的前提**。LVGL 的 `lv_font_t` 设计把 3 个函数指针 + 1 个 `dsc` 当作"虚表"，位图字体和 TTF 字体是两种实现。**页面代码只跟 `lv_font_t *` 打交道**，不知道也不关心底层是谁。这种设计让我们在一轮迭代里换了 3 次字体方案，页面代码 `page_*.c` 只改过一次批量替换（那次是宏名的升级，不是类型变化）。

8. **子集化字符集选"GB2312 全集"是个平衡点**。小了（GB2312 一级 3755）人名和歌名容易缺字；大了（CJK Unified 20902）Flash 倍增。6763 字是现代汉语覆盖率和 flash 占用的甜点区。

## 后续可做

- **TTF 迁 spiffs 分区**：若未来想做"多字体动态切换"（比如用户在菜单里选字体风格），把 TTF 放 spiffs 分区 + 开 `LV_TINY_TTF_FILE_SUPPORT=y`，运行时切换不用重烧
- **字符集缩减**：采集实际 UI 运行 1 个月出现过的字，做成白名单，做一轮"按需子集化"，TTF 可能能降到 ~800 KB
- **cache_size 调优**：如果翻页时感觉到卡顿（说明工作集超 256 字），把 Tiny TTF 的 cache 开到 512 或 1024（PSRAM 无压力）
- **备选字体工作流**：霞鹜文楷的 `.bak` 保留着。如果未来想要楷体风格（比如儿童向 UI），把 Noto 移走，`.bak` 去掉后缀，重跑 `gen_font_subset.py` 即可——脚本会自动选到楷体，产物还是 `srhs_sc_subset.ttf`，不改代码
- **自动化 CI**：`tools/gen_font_subset.py` 可以挂进 CI（pip install fonttools + python 一行），让字体子集化成为构建的一部分，避免开发者漏跑脚本导致的"字体文件不一致"
- **`lv_tiny_ttf_destroy()` 调用**：目前字体生命周期 = 程序周期，从未调 destroy。架构上如果做动态字体切换要补这一环
