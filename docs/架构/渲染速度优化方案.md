    先理解：屏幕上的画面到底是怎么来的

  LVGL 里有个核心概念叫 screen（屏）。它不是物理屏幕，是一棵 LVGL 对象组成的树：

  screen (root)
   ├─ top_bar (panel)
   │   └─ back_btn (button)
   │       └─ "返回" (label)
   ├─ title (label)
   └─ list_root (panel)
       ├─ row_0 (panel)
       │   ├─ btn_C / btn_± / btn_% / btn_÷
       ...

  LVGL 内部维护一个全局变量叫 active screen，是这样的：

  lv_obj_t *active_screen;   // LVGL 内部的一个指针

  每隔 ~16ms（取决于刷新率），LVGL 干一件事：遍历 active_screen 那棵树，把每个对象画到屏幕上。

  lv_scr_load(some_screen) 这个函数实际上就一行有用的逻辑：把 active_screen 这个指针指向 some_screen。

  所以底层的洞察是：

  ▎ 一个 screen 树存在 ≠ 它被画到屏幕上

  只有 active_screen 那一棵会被画。其他的 screen 树就静静躺在内存里，LVGL 完全不管它——既不渲染，也不收事件，就是一坨数据。

  这个事实是整个优化的根基。

  ---
  再理解：为什么原来"组件一个个出现"

  原来的流程：

  1. 用户点 calc 按钮（active_screen = menu）
  2. router 立刻调 lv_scr_load(new_screen)，把空的新屏挂上去
     → 此时 active_screen 已经是新屏了，但里面只有顶栏和空 list_root
     → LVGL 下一帧就开始画这个"空白页"
  3. 脚本任务被唤醒，开始执行 calc.js
  4. 脚本调 sys.ui.createPanel('disp')，往队列塞一条命令
  5. UI 任务每 10ms 来 drain 32 条命令
  6. 命令一条条执行 → 一个个对象 add 到 active_screen 上
  7. LVGL 下一帧渲染时发现"诶？多了个 disp panel"，就把它画出来
  8. 又下一帧"诶？又多了个按钮"，画出来
  ...

  关键问题：对象树是边构建边渲染的。第 2 步那个"空白页"已经成为 active_screen，从那一刻起 LVGL 每 16ms
  就拍一次"快照"显示给用户。每拍一次快照都看到对象树多长出几个零件，于是用户就看到了"组件陆续出现"的过程。

  类比：你在装修房子，工人一边运家具进来一边客人在屋里看。客人看着空房 → 摆了沙发 → 又来了茶几 → 又来了电视 → 装修完。

  ---
  优化的核心思路

  ▎ 让所有家具在隔壁房间装好，装好后一秒钟把客人请过来。

  技术上对应：

  ▎ 新 screen 树先不要作为 active_screen。让脚本在它上面随便组装。组装完成那一刻，再把 active_screen 指针一下子切过去。

  LVGL 的"一帧只画一棵树"特性保证：客人（用户）看到的要么是装修前的 menu，要么是装修后的完整 calc，不会看到中间状态。

  这就是为什么你提的"两棵树并存没问题"——LVGL 本来就支持这件事，只是平时大家用 lv_scr_load 都是立刻切，所以感受不到。

  ---
  拼装到代码里：四个层次的工作

  要把这个想法落地，每一层都要做点配合，因为原来整套系统是"切屏即建即跑"的假设：

  第 1 层：page_router

  router 原本只有 switch(id) 一个方法，语义是 "销毁当前页 + 调 create() 建新页 + 切屏" —— 三件事捆死。

  我们加了一个新方法 commit_prepared(id, prepared_screen)，语义改成 "销毁当前页 + lv_scr_load(prepared_screen) +
  切屏"——把建新页这一步抠掉，让外面自己建好再交进来。

  router 的内部状态（current_page / current_screen）跟着 commit_prepared 也更新，所以下次返回 menu
  时它知道该销毁谁。这一层是为"两棵树并存"打开口子。

  第 2 层：page_dynamic_app

  这是真正干"在隔壁房间装修"的地方。新增一个 prepare_and_switch(name)：

  prepare_and_switch("calc"):
    1. 在内存里 lv_obj_create(NULL) 建一棵新 screen（不 lv_scr_load！）
    2. 把这棵树的根 list_root 注入到 dynamic_app_ui 模块
       → 脚本以后所有 createPanel 都长在这棵 off-screen 上
    3. 注册一个回调，告诉 ui 模块"等脚本搭完了通知我"
    4. 起一个 800ms 定时器兜底（万一脚本卡住）
    5. 启动脚本
    6. 函数立刻返回，menu 还在显示

  这个函数返回后，UI 任务继续正常驱动 LVGL 画 menu，用户毫无感觉。脚本在另一个任务里悄悄跑，build 出来的对象长在那棵 off-screen 上。

  为什么这一层要做状态机（IDLE / PREPARING / COMMITTED）？因为用户操作时间线变复杂了：

  - 用户点了 calc 又秒点 alarm —— 上一个 prepare 还没完，得先撤销
  - 用户点了 calc 又点了 System —— 也得撤销
  - 用户连点 calc 两次 —— 后一次得忽略

  没有状态机就没法判断"现在该做什么"。

  第 3 层：dynamic_app_ui（桥接层）

  脚本任务和 UI 任务之间通过一个命令队列通信：

  脚本任务: createPanel(...) → enqueue 一条命令
  UI 任务:  从队列里取命令 → 真正调 lv_obj_create

  我们要解决的问题是：UI 任务怎么知道"脚本搭完了"？

  观察一个事实：所有 4 个动态 app 的 JS 末尾都长这样：

  rerender();  // build 整棵 UI
  sys.ui.attachRootListener("appRoot");  // 注册根事件监听

  attachRootListener 必然是脚本搭完 UI 之后才调。所以这条命令就是天然的"build 完成"信号。

  我们在桥接层加一个 ready 回调：drain 处理完一条 ATTACH_ROOT_LISTENER
  命令后，触发一次回调然后清空。这一行触发就让宿主页知道"装修完了，可以请客人了"。

  为什么是一次性？因为脚本理论上可能调多次（虽然现在没人这么做），重复触发会让宿主页重复 commit 一棵已经是 active 的
  screen，状态会乱。

  第 4 层：page_menu

  只是把 4 个动态 app 入口的 click 回调改成调 prepare_and_switch。

  但还有个细节：用户在 prepare 期间点了别的菜单项（比如 System），原本的 page_router_switch(PAGE_SYSTEM) 不会感知到"还有个 prepare
  在跑"。所以我们在所有跳转入口前加一行 cancel_prepare_if_any()——它只在 PREPARING 状态有效，其他时候是 no-op。

  ---
  整个流程串起来再看一遍

  T+0ms     用户点 calc
            → on_calc_clicked → prepare_and_switch("calc")
            → 后台建 off-screen (10ms 不到)
            → set_root(off-screen 的 list_root)
            → 注册 ready_cb
            → 起 800ms timer
            → dynamic_app_start("calc")  // 通知脚本任务
            → 函数返回，control 回到 UI 主循环
  T+0~80ms  UI 任务每 10ms 循环：
            - drain 命令队列（脚本陆续灌命令进来）
            - LVGL 渲染 active_screen ←—— 还是 menu，用户看到 menu
            - 命令执行：在 off-screen 上不停 lv_obj_create / set_text / ...
            - LVGL 不管 off-screen，它不参与渲染
  T+80ms    脚本调 attachRootListener
            → 命令入队
  T+90ms    UI 任务 drain 到这条命令
            → 在 off-screen 的 root 上挂 LVGL event_cb
            → 触发 ready_cb
            → 调 page_router_commit_prepared(PAGE_DYNAMIC_APP, off_screen)
            → router: destroy menu (lv_obj_del) + lv_scr_load(off_screen)
            → active_screen 指针突然指向那棵已经长好的树
  T+91ms    LVGL 下一帧渲染：用户突然看到完整 calc

  用户体验：menu → ... → menu → 完整 calc。没有空白页，没有组件冒出来的过程。

  ---
  几个值得记住的"底层规律"

  1. active 不等于存在：很多 GUI 系统都有这种设计（浏览器的 hidden tab、iOS 的后台 app）。利用这个特性可以做"预加载"。
  2. 跨任务通信用队列 + 标记位：脚本任务和 UI 任务不能直接互调（线程安全），用命令队列单向流。"事件完成"这种信号通过 callback +
  一次性标志位传，不要做长连接式订阅。
  3. 状态机比 bool 更安全：当一个对象有"准备中 / 就绪 / 已切走"等多种状态时，用 enum 而不是几个 bool。bool
  之间的合法组合靠人脑维护一定会忘，enum 把非法状态从语法层就排除了。
  4. 协议不变是金子：底层 page_router 加了新 API 但没改老 API 的语义。原来用 page_router_switch
  的所有页面零改动。这是非破坏性扩展的典范。
  5. 超时兜底是异步系统的良心：任何"等回调"的逻辑都要配超时。否则脚本里写错一行，整个 UI 永远卡 menu。800ms
  这个数字不是拍的——它略大于最复杂的 app 的 build 时间，是"绝大多数情况不触发，偶尔兜底用"的设定。

  ---
  类比总结

  把这套机制类比一下你就忘不了：

  - 原来：客人来了直接进毛坯房看着工人装修 = 切屏后再 build UI
  - 现在：客人在客厅等，工人在隔壁房间装修，装完一秒钟把门打开请客人过去 = prepare + commit

  LVGL 给的天然支持就是"隔壁房间存在不影响客厅显示"。我们做的工作就是搭一套门铃系统（ready 回调）+ 门卫（状态机 + 超时）+
  领位员（commit_prepared）。
