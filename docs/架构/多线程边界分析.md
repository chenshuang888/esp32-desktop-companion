  二、各线程接口面（主调入口）

  ui_task（消费者模型）

  被动模式：每 10ms 跑一圈，按顺序 drain 所有 manager 的 pending 队列：

  dynamic_app_ui_drain(32)      ← script_task 的 UI 命令
  time_manager_process_pending()  ← BLE host
  weather_manager_process_pending()
  notify_manager_process_pending()
  media_manager_process_pending()
  system_manager_process_pending()
  page_router_update()
  lvgl_port_task_handler()

  对外暴露的写入接口（其它线程可调）：
  - 各 manager 的 xxx_xxx() push API（背后 xQueueSend(timeout=0)）
  - dynamic_app_ui_enqueue_*（背后 xQueueSend）

  script_task（生命周期 + 消费）

  被动 + 主动混合：
  - xTaskNotifyWait(START/STOP) 等生命周期信号
  - 跑起来后周期性 drain dynamic_app_drain_ui_events_once / drain_ble_inbox_once / run_intervals_once

  对外暴露：
  - dynamic_app_start(name) / dynamic_app_stop()：xTaskNotify，线程安全
  - dynamic_app_ui_push_event / dynapp_bridge_push_inbox：xQueueSend，线程安全

  dyn_fs（纯消费）

  完全被动：xQueueReceive(portMAX_DELAY) 循环，分发到 dispatch_*。

  对外暴露：8 个 dynapp_fs_worker_submit_*，全部 xQueueSend(timeout=0)。

  nimble_host（BLE 协议栈）

  NimBLE 自己跑 host task。所有 GATT access_cb / on_subscribe / on_disconnect 都在它身上。

  对外暴露：
  - ble_gatts_notify_custom（NimBLE API，自带锁，跨线程安全）
  - ble_driver_get_conn_handle / ble_driver_is_connected（封装查询，本质是读 NimBLE 状态）

  四、潜在问题清单（按严重度排序）

  🟡 中：s_current_id 跨线程读写无 barrier

  dynamic_app_registry_set_current() 被外部线程调用（菜单 / page_dynamic_app），dynamic_app_registry_current() 被
  script_task（natives）和 fs_worker（running_check）读。

  static char s_current_id[16] = "";

  void dynamic_app_registry_set_current(const char *app_id) {
      strncpy(s_current_id, app_id, sizeof(s_current_id) - 1);
      ...
  }

  风险：strncpy 不是原子的，理论上 fs_worker 的 running_check 可能读到"半新半旧"的字符串（前几字节是新名
  后几字节是旧名）。最坏情况：误判一个 app 在跑，拒删。

  实际触发：极低，set 通常发生在 start 之前，那时 app 没在跑、菜单也没人长按。但不是零。

  建议修复：用 mutex 包一下，或者 atomic 指针交换（双 buffer）。优先级低但要记账。

  🟡 中：s_rt.app_running 跨线程读

  extern dynamic_app_runtime_t s_rt;   // 内部状态

  void dynamic_app_start(const char *name) {
      ...
      if (s_rt.app_running) continue;   // 在 script_task 读
  }

  实际上 app_running 只在 script_task 内修改（main loop 的 set true / set false），外部线程通过 xTaskNotify 间接驱动 ——
  所以被读但不被写。是安全的。但需要文档化"app_running 是 script_task 私有"，避免后续加 feature 时被误改。

  🟡 中：register_status_cb 没有反注册

  esp_err_t dynapp_upload_manager_register_status_cb(upload_status_cb_t cb);
  // 没有 unregister_status_cb

  page_menu 注册的 on_upload_status 是文件作用域 static，菜单页 destroy 时不会清。如果未来菜单页 destroy 后又有 BLE 上传完成，cb
  仍会被调用。

  当前安全：cb 内只 set 一个 static volatile bool，菜单页不在时下一次 create 会重置 dirty。但如果未来 cb 实现里 access 了菜单页的
  lv_obj_t —— 立刻空指针。

  建议：要么加 unregister，要么在文档约束 cb 必须 file-static + 只设 flag。

  🟢 低：page_menu 的 s_dirty 多写者

  写者：BLE host（manager status 广播）+ fs_worker（本地删除 done cb）。
  读者：ui_task（page_menu_update poll）。

  bool 单字写入对 ESP32-S3 (32-bit) 是原子的，没问题。但多写者时，BLE 删 + 本地删撞在一起会有"重复触发一次
  rebuild"，没害但浪费几十毫秒。

  🟢 低：upload_manager 的 status_cb 在两个线程被调

  // notify(...) 既会在 BLE host task 上调（CHUNK ack 同步路径）
  //          也会在 fs_worker task 上调（commit/delete/list done cb）
  for (int i = 0; i < s_cb_count; i++) {
      s_cbs[i](op, r, seq, name, extra, list_buf, list_len);
  }

  BLE service 自己的 cb（on_manager_status → ble_gatts_notify_custom）线程安全，没问题。
  page_menu 的 cb（设 s_dirty）也安全。

  但两个线程并发调 cb 这件事没在 .h 文档里写清楚。新加观察者的人可能不知道，里面写了 LVGL 调用就完蛋。

  建议：.h 里加一行 "cb may be invoked on either BLE host task or fs_worker task; never call LVGL"。

  🟢 低：app_ui.c 的死 ui_task

  app/app_ui.c:299 起了一个 ui_task 但没人调，只有 app_main.c 那个生效。建议直接删掉 app_ui.c 里的，避免未来有人误以为有第二个 UI
  线程。

  六、扩展性建议

  即将加 WiFi 时的注意

  WiFi 也会拉起几个 ESP-IDF 内部 task（tiT, wifi, lwip 等）。业务侧大概率会有：
  - 一个 wifi_manager task（订阅事件 / 重连策略）
  - 也许一个 http_client task（异步请求）

  复用现有模式：跟 time/weather/notify 一样，wifi_manager 暴露 wifi_manager_process_pending(ui) +
  wifi_manager_register_event_cb。事件队列消费在 ui_task。

  即将加 sys.http API 时

  JS 业务想发 http 请求：千万不要让 script_task 直接 esp_http_client_perform（同步阻塞几秒）。

  正确姿势：参考 fs_worker，新建 services/dynapp_http_worker，独立 task + queue + done cb。sys.http.get(url, cb) JS 入队、cb
  通过新加一个 dynapp_http_inbox 队列回到 script_task drain（保证 cb 在 JS 引擎线程调）。

  警惕"再加一个 task 就是了" 的陷阱

  每个 task 4-8KB stack + 1KB context。N16R8 PSRAM 16MB 不缺，但调试复杂度是 O(N²)：4 个 task 有 6 条潜在通信路径，5 个 task 有 10
  条。继续加之前先问"能不能复用现有 worker 的队列"。

  必做：补一份 docs/线程契约.md

  每个 task 一节，写清：
  - 角色 / 创建位置
  - 哪些 API 可以在我之外调（生产者）
  - 哪些 API 我会调出去（消费者）
  - 我持有的私有状态（不准外部触碰）
  - 我消费/产出的事件类型

  未来加新功能的人查这一份就行，不用读代码猜。这也跟你 spec 库里「BLE 跨层语义有契约」的硬规则同向。
