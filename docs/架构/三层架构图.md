三层铁律

  第一层：VDOM.xxx —— 懂树

  ┌───────────────────────────────┬──────────────────────────────┬──────────────────────────────────────────┐
  │             函数              │             干啥             │                "懂树"在哪                │
  ├───────────────────────────────┼──────────────────────────────┼──────────────────────────────────────────┤
  │ VDOM.h(type, props, children) │ 造一个 vnode（带子节点描述） │ 一造就是一棵子树                         │
  ├───────────────────────────────┼──────────────────────────────┼──────────────────────────────────────────┤
  │ VDOM.mount(node, parentId)    │ 把 vnode 树整棵挂上去        │ 递归挂所有 children                      │
  ├───────────────────────────────┼──────────────────────────────┼──────────────────────────────────────────┤
  │ VDOM.destroy(id)              │ 销毁 vnode 树                │ 递归销毁所有 children + 摘掉父节点的引用 │
  ├───────────────────────────────┼──────────────────────────────┼──────────────────────────────────────────┤
  │ VDOM.render(tree)             │ 比对新旧两棵树并增量更新     │ 走遍整棵树 diff                          │
  ├───────────────────────────────┼──────────────────────────────┼──────────────────────────────────────────┤
  │ VDOM.dispatch(id, type, ...)  │ 沿 _parent 链冒泡事件        │ 知道父节点是谁                           │
  ├───────────────────────────────┼──────────────────────────────┼──────────────────────────────────────────┤
  │ VDOM.set(id, patch)           │ 局部 patch 单节点            │ (这个是例外，是"不懂树"的精确打击)       │
  ├───────────────────────────────┼──────────────────────────────┼──────────────────────────────────────────┤
  │ VDOM.find(id)                 │ 按 id 查 vnode               │ 在 nodes{} 字典里查                      │
  └───────────────────────────────┴──────────────────────────────┴──────────────────────────────────────────┘

  特征：参数里要么是 vnode（一棵树），要么是 id（树里的一个节点）；内部通常会递归操作多个节点或沿 _parent 链跳跃。

  只在 JS 内存里跑，不直接操作 LVGL（除了底下调原语那一步）。

  第二层：sys.xxx —— 不懂树，只接受 id 字符串和原始值

  ┌──────────────────────────────────────┬──────────────────────┬───────────────┐
  │                 函数                 │         干啥         │     参数      │
  ├──────────────────────────────────────┼──────────────────────┼───────────────┤
  │ sys.ui.createPanel(id, parentId)     │ 创建一个 panel       │ 都是字符串 id │
  ├──────────────────────────────────────┼──────────────────────┼───────────────┤
  │ sys.ui.setText(id, text)             │ 改一个 label 的文字  │ id + 字符串   │
  ├──────────────────────────────────────┼──────────────────────┼───────────────┤
  │ sys.ui.setStyle(id, key, a, b, c, d) │ 改一个对象的某个样式 │ id + 整数     │
  ├──────────────────────────────────────┼──────────────────────┼───────────────┤
  │ sys.ui.destroy(id)                   │ 删一个对象           │ 一个 id       │
  ├──────────────────────────────────────┼──────────────────────┼───────────────┤
  │ sys.ui.attachRootListener(id)        │ 挂事件 listener      │ 一个 id       │
  ├──────────────────────────────────────┼──────────────────────┼───────────────┤
  │ sys.app.saveState(json)              │ 存数据               │ 字符串        │
  ├──────────────────────────────────────┼──────────────────────┼───────────────┤
  │ sys.log(text)                        │ 打日志               │ 字符串        │
  └──────────────────────────────────────┴──────────────────────┴───────────────┘

  特征：参数全是原始值（字符串 / 整数）。只描述"对一个 id 做一个动作"，不知道这个 id 是谁的儿子谁的父亲、不知道它有几个 children。

  调用流程：JS 调 sys.xxx → 把"动作"塞进 cmd queue → 立刻返回。不等结果——这就是你说的"队列传递"。

  第三层：C 侧 drain —— 真正干活的人

  UI 线程从 cmd queue 拉一条 cmd → 查 registry 找到 lv_obj_t* → 调 LVGL API 真改屏幕。

  JS:  sys.ui.setText("clockMain", "12:34:56")
         │ (入队 cmd: type=SET_TEXT, id=clockMain, text="12:34:56")
         │ (立刻返回)
         │
     ───── 队列 ─────
         │
         ▼
  C:   drain 拉到这条 cmd
         │
         ├─ registry_find("clockMain") → lv_obj_t* clockMainObj
         └─ lv_label_set_text(clockMainObj, "12:34:56")
                                            ↓ 屏幕上文字真变了

  你的提炼可以再加一句

  你说的两条：
  - ✅ VDOM.xxx 操作 VDOM 树
  - ✅ sys.xxx 只能按 id 操作具体对象 + 走队列

  我再加一条让它彻底清晰：

  ▎ VDOM.xxx 调用之后效果立刻在 JS 内存里生效；sys.xxx 调用之后效果异步生效，得等 UI 线程下一次 drain。

  举例：

  VDOM.set("dispText", { text: "12.5" });
  // 这行执行完：
  // - JS 内存里 nodes["dispText"].props.text 已经是 "12.5" ✓
  // - 屏幕上还是旧文字 ✗ （cmd 还在队列里没被消费）

  // 几毫秒后，UI 线程跑 drain，屏幕才真变

  这就是为什么 VDOM.find("dispText").props.text 立刻返回 "12.5" 而不是去问 LVGL——JS 看到的是"逻辑真相"，LVGL
  上的是"物理真相"，两者中间隔着一个队列的延迟。99% 情况下你感觉不到这个延迟，但理解它存在很重要。
