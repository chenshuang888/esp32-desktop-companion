## 一、设备端固件（esp32-desktop-companion / esp32-companion-firmware）

### 1.1 工具链与硬件契约

- **ESP-IDF**: README 自述 v5.4.3+，`main/idf_component.yml:6` 显式锁 `makgordon/esp-mquickjs ^0.2.1`、`lvgl/lvgl ^9.0.0`、`espressif/esp_lcd_touch_ft5x06 ^1.0.0`、`joltwallet/littlefs ^1.14.0`。NimBLE 走 IDF 自带（`CONFIG_BT_NIMBLE_ENABLED=y`，4.2 协议）。
- **目标芯片**: ESP32-S3 N16R8（Xtensa LX7 双核 240 MHz）。
- **Flash/PSRAM**: 16 MB QIO 80 MHz Flash + 8 MB Octal 80 MHz PSRAM。`sdkconfig.defaults:64-89` 详细注释了 N16R8 的关键陷阱——OCT_FLASH 必须 OFF，否则与 OCT PSRAM 时序冲突 boot 循环；`SPIRAM_MALLOC_ALWAYSINTERNAL=16384` + `SPIRAM_MALLOC_RESERVE_INTERNAL=32768` 实现"小分配走 internal、大分配落 PSRAM"，并 `CONFIG_BT_NIMBLE_MEM_ALLOC_MODE_EXTERNAL=y` 把 BLE 堆赶到 PSRAM。
- **LCD/触摸**: ST7789 240×320 SPI@40 MHz RGB565（`drivers/board_config.h:6-25`）；FT5x06 I2C@400 kHz；`BOARD_LVGL_BUFFER_LINES=40` → 双缓冲 240×40×2 字节 ≈ 19 KB 单缓冲，`heap_caps_malloc(MALLOC_CAP_DMA | MALLOC_CAP_INTERNAL)` 保证 DMA。引脚：LCD SCK=12 MOSI=11 CS=10 DC=9 RST=8 BL=14；TOUCH SCL=18 SDA=17 RST=15 INT=16。
- **分区表（`partitions.csv`）**: `nvs 24KB` + `phy_init 4KB` + `factory 6 MB`（含嵌入 ~3.4 MB TTF 子集）+ `storage 9.9 MB LittleFS` (subtype=0x83，注释明确这是私有 magic，从 SPIFFS 切过来必须 erase-flash 一次)。

### 1.2 启动顺序（`main/main.c`）

`main.c:46-111` 是非常工整的"先存储后服务后 UI"链：

1. `persist_init` / `backlight_storage_init` / `time_storage_init` —— 先把 NVS 拉起来，所有后续模块都可以读 NVS。
2. `fs_littlefs_init` → `dynapp_script_store_init` → `dynapp_fs_worker_init` —— LittleFS 挂载，启动一个**串行写入 worker task**（`storage/littlefs/dynapp_fs_worker.c`），任何线程都能非阻塞 submit 写任务，避免多线程写 LittleFS 的并发问题。
3. 恢复上次保存的系统时间（`time_storage_load_last`），失败回落到硬编码 2026-04-17。
4. 各 manager 先 init（time / weather / notify / media / playlist / system），它们要在 BLE 回调到来之前就绪，否则 push 队列没人接。
5. `ble_driver_nimble_init` 先拉 NimBLE 协议栈（`drivers/ble_driver.c:18` 设备名 `ESP32-S3-DEMO`），随后每个 service 自管自己的 GATT 表注册（`time_service_init` ... `dynapp_upload_service_init`），最后 `ble_driver_nimble_start` 起 host task。
6. `dynapp_mailbox_init` 离线消息总线（动态 App JS task 不在跑时把 BLE 消息归档到 NVS，下次启动回灌）。
7. `app_main_init` 起 UI（`app/app_main.c:111`）：8 KB 栈，pin 在 core 1，10 ms tick；每 tick 一次性 drain 32 个 dynamic_app UI 命令、process pending、`device_stats_tick`（1 Hz）、`app_router_tick`、`lvgl_port_task_handler`、`notify_manager_tick_flush`、`time_storage_tick_save`。

### 1.3 LVGL 与字体（关键工程亮点）

- LVGL 9.x（`^9.0.0`），`drivers/lvgl_port.c:62-80` 双缓冲 + LVGL 9 新 API（`lv_display_create` / `lv_display_set_flush_cb`）。
- 字体方案非常讲究（README §"中文字体"）：放弃 LVGL 自带 `LV_FONT_SOURCE_HAN_SANS_SC_*_CJK`（仅 1118 字），改用 **Tiny TTF 运行时渲染** + 子集化：
  - `tools/scripts/gen_font_subset.py` 扫源码提取真正用到的汉字 → `srhs_sc_subset.ttf`（~3.4 MB）
  - `tools/scripts/gen_icons_subset.py` 生 Material Symbols 图标子集（~25 KB）
  - 通过 IDF `EMBED_FILES` 嵌进 factory 镜像
  - 启动时 `lv_tiny_ttf_create_data_ex` 创建多组字体（text/title/huge/icon24/icon36），fallback 链挂回 `lv_font_montserrat_14/20/24` 解决 LVGL 内置图标
  - `CONFIG_LV_USE_CLIB_MALLOC=y` 让大块 glyph cache 自动落 PSRAM
- LVGL FS Driver 借道 esp_littlefs VFS（`CONFIG_LV_USE_FS_STDIO=y` + `LV_FS_STDIO_LETTER=65`/'A'），让 `lv_image` 能直接 `A:/littlefs/apps/<id>/assets/foo.bin` 读图。

### 1.4 分层架构（main → app → services → drivers + manager 单写）

```
main → app → app_router/sub_router
            → services/*_service.c (NimBLE host 线程)
                ↓ push 入队
              services/manager/*_manager.c (UI 线程独占消费 + 落 NVS)
            → drivers/*.c
            → dynamic_app → dynapp_bridge / dynapp_upload services
                            → scripts/<app>/main.js
```

两条铁律（README 显式重申）：
1. **services → drivers 单向依赖**。BLE 回调从不阻塞、从不访问 LVGL、从不写 NVS。
2. **LVGL 不是线程安全的** → JS 脚本线程**永远不直接调 LVGL**，只能 enqueue 指令；UI 线程 drain。

各目录职责：
- **`drivers/`** (587 LOC C/H) — `board_config.h`、`lcd_panel.c`、`touch_ft5x06.c`、`lvgl_port.c`、`ble_driver.c`(NimBLE 协议栈封装 + GAP + SUBSCRIBE 分发表 `s_sub_cbs[8]`，service 自注册自分发)。
- **`framework/`** (458 LOC) — `app_router.c/h`(顶层 app 切换) + `sub_router.c/h`(单 app 内多级页)。
- **`services/`** (3225 LOC, 不含 manager) — 8 个 service：time / weather / notify / media / system / dynapp_bridge / dynapp_upload，外加 manager/ 子目录里 11 个 `*_manager.c`(time/weather/notify/media/playlist/system/device_stats/battery_sim/dynapp_mailbox/dynapp_upload_manager 等)。
- **`storage/`** (1596 LOC) — `nvs/persist.c` 等 NVS 适配 + `littlefs/fs_littlefs.c` + `dynapp_script_store.c` + `dynapp_fs_worker.c`(异步串行写)。
- **`app/`** (7569 LOC, 占整个固件 C 代码 43%) — `app_main.c`(UI task) + `app_fonts.c`(Tiny TTF 多字体初始化) + `weather_icons.c` + `fonts/`（嵌入 TTF）+ `icons/`（PNG 资源）+ **`apps/`** 9 个原生 App 模块（每个 app 独立目录，含 `<app>_app.c` 和 `pages/` 子目录）+ `ui/`（`app_shell_ui.c` / `ui_anim.c` / `ui_modal_card.c` / `ui_statusbar.c` / `ui_tokens.h` / `ui_widgets.c`）。
- **`dynamic_app/`** (3872 LOC C/H + 2714 LOC JS) — JS runtime 控制层。

### 1.5 BLE 协议（GATT）

读 `services/*` 把所有 UUID 和方向核完了：

| Service UUID | Char UUID | 方向 | Payload | 字节 |
|---|---|---|---|---|
| `0x1805` (CTS 标准) | `0x2A2B` | PC→ESP WRITE / NOTIFY | `<HBBBBBBBBB` 标准 CTS + 内部 1B seq | 10 / 1 |
| `8a5c0001-...` weather | `8a5c0002` | PC→ESP WRITE | `<hhhBBI24s32s` | 68 |
| 同上 | `8a5c000b` weather-req | ESP→PC NOTIFY | 1B seq | 1 |
| `8a5c0003-...` notify | `8a5c0004` | PC→ESP WRITE | `<IBB2x32s96s` | 136 |
| `8a5c0007-...` media | `8a5c0008` | PC→ESP WRITE | `<BBhhHI48s32s` | 92 |
| 同上 | `8a5c000d` media-btn | ESP→PC NOTIFY | `<BBH` (id/action/seq) | 4 |
| `8a5c0009-...` system | `8a5c000a` | PC→ESP WRITE | `<BBBBBBhIHH` | 16 |
| 同上 | `8a5c000c` system-req | ESP→PC NOTIFY | 1B seq | 1 |
| `8a5c00ee-...` dynapp upload | (rx + status) | PC→ESP WRITE / ESP→PC NOTIFY | 自定义封包 | ≤200 |
| `a3a30001-...` dynapp bridge | `a3a30002` | PC→ESP WRITE | 透传 JSON | ≤200 |
| 同上 | `a3a30003` | ESP→PC NOTIFY | 透传 JSON | ≤200 |

UUID 在 `services/dynapp_bridge_service.c:29-44`、`services/dynapp_upload_service.c:26-34`、`services/media_service.c:15-27`、`services/notify_service.c:12-18`、`services/system_service.c:13-25`、`services/weather_service.c:13-26`，CTS 在 `services/time_service.c:164-167`。8a5c 段是业务私有 base、a3a3 段专给动态 App 桥（视觉区分）。

**反向请求模式**：每个 service 自管一个 1B seq NOTIFY char。订阅上升沿或页面 enter 触发一次空 notify，PC 收到后用同 service 的 WRITE char 回写数据。media-btn 4B 包格外承担"屏上按钮 → PC 媒体键"事件（id=0 Prev / 1 PlayPause / 2 Next）。

`drivers/ble_driver.c:22-29` 是 8 槽订阅回调表，service 自己在 init 阶段调 `ble_driver_register_subscribe_cb` 注册，GAP `BLE_GAP_EVENT_SUBSCRIBE` 触发时遍历分发——这种"驱动只管协议栈、service 自注册"是典型的**反向依赖**模式。

### 1.6 内建原生 App（`app/apps/`）

9 个目录（README 说 7~9 个，实际 9 个）：lockscreen / launcher / clock / weather / notifications / music / system / settings / dynapp_host。每个目录两文件 + `pages/` 子目录（用 `sub_router` 管多级页）。最复杂的是 `clock/` 4 子页（alarms / timer / stopwatch / world）和 `music/`（list + detail 双页 + 媒体键回写）。`dynapp_host` 是关键宿主——见 1.7。

### 1.7 动态 App 平台（项目核心）

#### 引擎与分层
3 个 .c 文件组成：
- **`dynamic_app.c`** (`dynamic_app/dynamic_app.c:1-31`) — 控制层。`script_task` 主循环：阻塞等 `NOTIFY_START` → 清 intervals 表 → 清 inbox → mailbox replay → `dynamic_app_runtime_start_app` → tick 循环（每轮 drain UI 事件 + run intervals + `vTaskDelay≤50ms`）→ `NOTIFY_STOP` 回到等待。
- **`dynamic_app_runtime.c`** — 引擎层。`JSContext` 创建/销毁、bind 全局、eval prelude.js + app/main.js。
- **`dynamic_app_natives.c`** — API 层，所有 `sys.*` native 函数。

UI 桥独立 3 个文件：`dynamic_app_ui.c`（命令队列 drain）、`dynamic_app_ui_registry.c`（id→lv_obj 映射）、`dynamic_app_ui_styles.c`（style key 映射）。

JS 引擎是 **esp-mquickjs 0.2.1**（QuickJS 的 ESP-IDF 精简移植），仅支持 ES5（`docs/动态app_包格式规范.md:67`：禁箭头/let/const/class/模板字符串/Promise/原生 setTimeout）。

#### `prelude.js`（**1005 行**，标准库）
开机自动 eval 一次，所有动态 App 共享。提供 4 大子系统（前面 Read 过 `dynamic_app/scripts/prelude.js`）：

1. **VDOM** (`prelude.js:19+`) — `h(type, props, children)` + `mount` / `find` / `set` / `destroy` / `dispatch` / `render`(diff)。映射表 `FONT_MAP / ALIGN_MAP / FLEX_ALIGN_MAP / LONG_MODE_MAP / TEXT_ALIGN_MAP` 把 prop 翻译成 native 枚举。事件委托：在 root 上 `attachRootListener`，被点击的子对象 id 入队，由 `sys.__setDispatcher(VDOM.dispatch)` 注册的回调统一分发。
2. **UI 组件库** — `UI.screen / title / card / kvRow / listRow / iconBtn / pillBtn / badge / statusBar / divider / hitZone / modal / toast / fadeIn / swipeExit`。`SIZE_CONTENT` 常量。
3. **Router** (`prelude.js:775-974`) — 多级页面栈纯 JS 实现，**MAX_DEPTH=4**。`define / start / push / replace / pop / popTo / current / depth / onEnter / onLeave`。push 不 destroy 旧页（保留滚动位置 / canvas / interval），只切 hidden flag。屏底自动 mount 一个 30px 的 `_routerSwipe` panel，上滑 ≥30px 触发 pop；栈深=1 时事件穿透到宿主层退出 app。
4. **`makeBle("appName")`** + `setTimeout / clearTimeout`（基于 setInterval 包一层）。

#### `sys.*` Native API（`dynamic_app_natives.c` grep 出来的）
- `sys.log`
- `sys.ui.*` — `setText / createLabel / createPanel / createButton / createImage / setImageSrc / setStyle / attachRootListener / __setDispatcher / destroy / modal / toast / fadeIn`
- `sys.time.*` — `now / parts / format / uptimeMs / uptimeStr`
- `setInterval / clearInterval`（顶层全局，不在 sys 下）
- `sys.app.*` — 持久化 `saveState(json) / loadState() / eraseState()`，落 NVS namespace `dynapp_<id>`
- `sys.ble.*` — `send / onRecv / isConnected`（透传 a3a3 桥）
- `sys.fs.*` — `read / write / exists / remove / list`，**沙箱在 `/littlefs/apps/<current_app>/data/`**，没 current_app 就直接 throw
- `sys.canvas.*` — `create / fill / setPixel / line / saveTo / loadFrom`（像素级绘图 + 与 LVGL bin 互转）
- 常量：`sys.symbols / sys.style / sys.align / sys.font / sys.tokens / sys.icons / sys.size`

事件队列里所有命令都"取参 → 入队 → 立即返回 bool"，**绝不直接调 LVGL**——这是动态 App 平台多线程安全的根基。

#### dynapp_bridge BLE 透传管道（`services/dynapp_bridge_service.c`）
- 单一 GATT service，开机注册死，运行时不增删。
- C 不解析 payload，纯 byte 搬运。Inbox 8 槽 FreeRTOS Queue。满则丢最老（PC 推无 backpressure）。
- App 切换时（`dynamic_app.c:118` `dynapp_bridge_clear_inbox`）清空 inbox 避免串台。
- 业务层用 JSON：`{"from":"app_id","type":"move","body":{"r":3,"c":4}}`，单条 ≤200 B（应用层自分包）。

#### dynapp_upload 协议（`tools/dynapp_uploader/protocol.py`）
小端二进制帧 `[1B op][1B seq][2B len][payload]`：
- `0x01 START` — 39B payload = 31B path + 4B total_len + 4B crc32
- `0x02 CHUNK` — 4B offset + ≤MAX_CHUNK data
- `0x03 END` — 空
- `0x10 DELETE` — 15B app_id（删整 app 目录）
- `0x11 LIST` — 空（status notify 用 NUL 分隔字节流回 app 名列表）
status notify 反向：`[1B op_echo][1B result][1B seq_echo][1B reserved][payload]`，CHUNK OK 携带 next_expected_offset。

#### 切屏体验（关键工程亮点）
直接同步切到动态 App 用户能看到组件一个个冒出。`app/apps/dynapp_host/pages/page_dynapp_host.c` 的方案：
1. `dynapp_host_prepare_and_enter(app_name)` 立即返回不切屏
2. 后台建 off-screen 对象树，让 JS 在隐藏 root 上 build UI
3. JS 调 `sys.ui.attachRootListener` 触发 ready 回调，路由 `commit_prepared` 瞬间换屏
4. 800 ms 兜底超时强切（防 JS 死循环）

#### 离线消息总线 `dynapp_mailbox`（`services/manager/dynapp_mailbox.c`）
JS task 不在跑时把 BLE 消息归档到 NVS，启动对应 app 时回灌进 inbox。让动态 App 重启后能看到自己掉线期间的消息（如 notif 推送）。

### 2.2 companion 入口与调度（`companion/__main__.py`、`core.py`）

`__main__.py:16-57` 构建链：
- `EventBus`(`bus.py`) 单例
- `Companion(device_name, bus, device_address?)` 唯一 BleakClient + provider 注册器
- `PluginManager(bus, tx_func, is_connected)`：扫两个目录（仓库 `tools/plugins/` + 用户 `%APPDATA%/esp32_companion/plugins/`）→ importlib 加载 `Plugin` 子类
- 注册 7 个 provider：`TimeProvider / WeatherProvider / NotifyProvider / SystemProvider / MediaProvider / BridgeProvider(plugin_manager=pm) / UploadProvider`
- `bus.on("connect"/"disconnect")` 转发给所有插件

- **一致性检查**(`scanner.ts:223-247`):
  - JS 用 `makeBle / ble.send / ble.on` 但 manifest.requires.ble=false → warn
  - JS 用 BLE 但包内没 plugin/ → **danger**(PC 端没人响应)
  - manifest.requires.plugin=true 但没 plugin/ → danger
  - 有 plugin/ 但缺 plugin/plugin.py → warn
- **rollupStatus**: 任一 danger → danger，否则有 issue → warn，否则 safe

`lib/ai-review.ts:21-39` AI 审核 system prompt 严格要求 JSON 输出 schema（verdict / summary / consistency / risks[]）。`reviewWithAi` 走 OpenAI SDK 通用客户端，默认 `https://api.deepseek.com` + `deepseek-chat`，可在 admin/settings 切供应商。`pingAi` 是"测试连接"按钮的健康检查。

## 四、动态 App 包格式（`.mpkg`，贯穿三端）

### 4.1 物理布局（zip）

```
foo.mpkg (zip)
├── manifest.json     必需 {id,name,version,description?,icon?,color?,entry?,requires?}
├── main.js           必需，ES5
├── icon.png          可选（市场图标，不是设备图标）
├── README.md         可选（市场说明）
├── assets/           可选，单层
│   ├── *.bin
│   └── ...
└── plugin/           可选，PC 端 Python 插件
    ├── plugin.py     入口，继承 Plugin 基类
    └── gui_page.py   可选 CTk 页面
```

### 4.2 设备端解包（`backend/src/lib/scanner.ts:139-196` + 设备端 fs_worker）

PC 端 `installer.py` 把 `.mpkg` zip 解出 manifest+main.js+assets，写临时目录后调 `UploadProvider.upload_app_pack` 走二进制 OP_START/CHUNK/END/DELETE 协议；设备端 `dynapp_upload_service.c` + `dynapp_upload_manager.c` + `dynapp_fs_worker.c` 接收，逐文件 `.tmp → rename` 原子提交到 `/littlefs/apps/<id>/{main.js,manifest.json,icon.bin?,assets/*}`。launcher 通过 `dynamic_app_registry` 观察文件变化热刷新。

### 4.3 PC 端解包

`installer.py:59-118` `parse_mpkg` 抽 manifest+main.js+assets+plugin/+icon.bin+README，**zip-slip 双重防御**（包内路径 `..` + 解出后 `target.resolve().relative_to(plugin_root.resolve())`）。`plugin/` 解到用户级 `%APPDATA%/esp32-companion-user/plugins/<slug>/`，`registry.py` 记账文件清单便于卸载时只删登记过的文件（不动用户手改文件）。装完需要重启 companion 加载新插件（无 hot reload）。

### 4.4 设备端 ↔ PC 端 通信契约
JS 业务用 `makeBle("foo")` 拿 `{send, on, onAny, onError, isConnected, appName}`，`send(type, body)` 包成 `{from:"foo", type, body}` JSON over a3a30002 写到设备；PC 端 `BridgeProvider` 收 a3a30003 派 `PluginManager.dispatch_message` → 按 `bind_app="foo_pkg"` 派给 `FooPlugin.on_message`。反向 `self.tx(type, body)` → `bus("bridge:tx")` → `BridgeProvider` 写回 a3a30002。

3. **55 篇按时序排列的工作日志**——每个里程碑都有决策依据和踩坑记录，对未来维护者极友好
4. **三套部署方案**（开发 hot-reload、Caddy 生产、宝塔生产）覆盖国内/国外/手熟/图形界面运维

7. 用户级 vs 仓库级**双插件目录**让市场装的 plugin 不污染源码仓
8. 发布版自动落 `%APPDATA%/esp32-companion-user/`，删目录即重置

### 6.5 不足 / 粗糙处（不回避）

- **零自动化测试**：没有 pytest / vitest / unity 测试，所有验证都靠工作日志里的"手测+串口"。
- **依赖闭源 LLM API（DeepSeek）做安全审核**——审核规则可控但 LLM 一致性靠不住，AI provider="disabled" 是默认 fallback。
- **包没签名**——任何能连 BLE 的人都能 push 任意脚本到设备，理论上 PC 端 SDK 改两行就能绕过 marketplace 直接装 banned 包。
- **ES5-only QuickJS** 限制业务作者心智负担，长远看应该升 ES2020。
- **整个项目无 CI/CD**：没有 GitHub Actions / 没自动 release，每次发布要手动 build + 上传 release。
- 旧版本字段（`icon.bin`）在文档里被显式标"已废弃"但代码兼容路径还在——历史包袱已经开始累积。
- esp32-desktop-companion 的 `.git/` 历史很热闹但 esp32-companion-firmware / esp32-companion-app / esp32-marketplace 三个发布仓都是 squash 后的"initial release"风格，**关联不上历史**——开源用户从发布仓追源很困难，必须看 esp32-desktop-companion。
- `.devcontainer/`、`.claude/`、`AGENTS.md`(Trellis 块)、`docs/Trellis工作流适配日志.md` 显示项目也大量用了 AI 工具开发，README 里没披露。
- **`docs/未完成的对话.md` / `docs/未总结出工作日志的对话记录.md`** 这两个文件名暴露了"工作日志是事后整理 AI 对话产出"的工作流——这不是缺陷，但读 README 时它隐而不彰。
